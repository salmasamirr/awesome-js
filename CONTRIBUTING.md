This project hosts JavaScript/TypeScript libraries under `packages/` and demo apps under `examples/`.  
The first package is `@awesome/charts` (Angular), but the structure supports **multiple frontend frameworks** (Angular, React, Vue, Svelte, etc.).

---

## Prerequisites

- Node.js 20+
- npm 10+
- Git
- Windows PowerShell (or any POSIX shell on macOS/Linux)

Install dependencies per workspace (there is no root `package.json`):

```bash
cd packages/charts && npm ci --no-audit --no-fund
cd ../../examples/charts && npm ci --no-audit --no-fund
```

## Development Workflow

1. **Build a library**

   ```bash
   cd packages/charts
   npm run build
   ```

   Output goes to `dist/charts` with bundles and typings.

2. **Run the example app**

   ```bash
   cd ../../examples/charts
   npm start
   ```

   Examples resolve local packages via TypeScript path mapping, e.g.:

   ```json
   {
     "compilerOptions": {
       "paths": {
         "@awesome/charts": ["../../dist/charts"]
       }
     }
   }
   ```

3. **Watch mode (optional)**
   - Re-run `npm run build` for library changes.
   - Run `npm run watch` in the example app for live reload.

## Coding Standards

- Angular libraries target Angular 20.x  
- Non-Angular libs can use React/Vue/Svelte with modern toolchains (Rollup, Vite, etc.)
- TypeScript ~5.9 preferred
- Explicit exports via `src/public-api.ts` (for Angular) or `index.ts` (for others)
- Clean code: meaningful names, early returns, minimal nesting
- Formatting: Prettier (printWidth 100, singleQuote)

## Commit Messages

Use conventional commits:

- `feat`: new feature
- `fix`: bug fix
- `docs`: documentation
- `refactor`: code change w/o behavior change
- `test`: add/update tests
- `build`: build process/config
- `chore`: maintenance

Examples:
- `feat(charts): add LLMService token configuration`
- `fix(charts): guard against empty series in echarts options`

Breaking changes â†’ include `!` and describe in PR.

## Branching & PRs

- Branch from `main`
- Keep PRs focused
- Include:
  - Summary (what/why)
  - Screenshots or GIFs (for UI changes)
  - Testing notes
  - Breaking change notes
- Ensure build passes:
  - `cd packages/<lib> && npm run build`
  - `cd examples/<lib>-example && npm run build` or `npm start`

## Testing

- Angular demos: `npm test` (Karma/Jasmine)
- Other frameworks: use their default test runners (Jest, Vitest, etc.)
- Add/update tests when changing behavior

## Adding a New Library & Example

1. Create a new library under `packages/<new-lib>`
   - Include `package.json`, build script, and entry (`public-api.ts` or `index.ts`)
   - Example Angular `ng-package.json`:

   ```json
   {
     "$schema": "../../node_modules/ng-packagr/ng-package.schema.json",
     "dest": "../../dist/<new-lib>",
     "lib": { "entryFile": "src/public-api.ts" },
     "allowedNonPeerDependencies": []
   }
   ```

   Example minimal `package.json`:

   ```json
   {
     "name": "@awesome/<new-lib>",
     "version": "0.0.1",
     "scripts": { "build": "ng-packagr -p ng-package.json" }
   }
   ```

2. Create a demo app under `examples/<new-lib>-example`
   - Configure `tsconfig.json` paths to map `@awesome/<new-lib>` to `../../dist/<new-lib>`
   - Import and use the library

3. Build & run:

   ```bash
   cd packages/<new-lib> && npm run build
   cd ../../examples/<new-lib>-example && npm start
   ```

## Release & Versioning (Maintainers)

- Each library versioned in its own `package.json`
- Follow semver
- Keep peerDependencies aligned with framework versions
- Do not commit `dist/` or `node_modules/`
